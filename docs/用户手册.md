# 使用手册

本手册面向第一次接触这个项目的使用者，目标是帮助你在 GitHub 仓库里快速理解：

- 这个工具能做什么
- 它适用于什么游戏
- 你需要准备什么
- 应该按什么顺序操作
- 常见问题应该怎么判断

## 1. 先确认适用范围

这个项目目前只支持 `Unity Mono` 游戏。

你可以把它理解成一个围绕下列目录结构工作的本地化辅助工具：

- 游戏根目录下有 `Game.exe`
- 同级存在 `Game_Data`
- `Game_Data` 下存在 `Managed`
- 有些游戏还会带 `MonoBleedingEdge`

这个项目当前不适用于：

- `Unity IL2CPP`
- 非 Unity 游戏
- 通用 OCR 抓屏翻译
- 任意 Windows 程序文本注入

如果目标目录里没有 `*_Data/Managed`，那它大概率就不是这个工具当前支持的对象。

## 2. 这个工具的核心思路

这个项目不是重新打包游戏资源，也不是修改所有游戏文本文件。

它当前的工作思路更接近：

1. 识别目标游戏是否为支持的 `Unity Mono`
2. 准备一套可用的翻译运行时目录
3. 把运行时里的必要文件安装到目标游戏
4. 为目标游戏生成 `AutoTranslator\Config.ini`
5. 启动本地翻译桥接服务
6. 让游戏运行时通过本地接口请求翻译结果
7. 把翻译结果缓存到本地，减少重复请求

## 3. 你需要准备什么

使用前至少要准备下面这些内容：

### 3.1 Python 环境

- Windows
- Python `3.12+`

### 3.2 一个受支持的游戏目录

目标游戏目录至少应该像这样：

```text
YourGame/
  YourGame.exe
  YourGame_Data/
    Managed/
```

### 3.3 一个运行时目录

运行时目录是工具安装时要复制到目标游戏里的那一套文件集合。

在这个项目当前实现里，一个可用的运行时目录通常包含：

```text
runtime/unity_mono/
  AutoTranslator/
  ReiPatcher/
  Managed/
```

如果你仓库里没有现成 `runtime/unity_mono`，那就需要通过“导入运行时”功能，从一个已经准备好的样例游戏目录中提取。

### 3.4 可选：翻译服务配置

如果你要使用大模型翻译或普通机翻，还需要准备对应接口配置，例如：

- API Base
- API Key
- 模型名

这些配置保存后会写入本地：

```text
.private_translator/settings.json
```

这是本地运行数据，不建议提交到仓库。

## 4. 安装项目

在仓库根目录执行：

```powershell
python -m pip install -e .
```

安装完成后，你可以使用：

```powershell
game-translator --help
```

也可以直接使用：

```powershell
python main.py --help
```

## 5. 推荐操作顺序

第一次使用时，建议严格按这个顺序走：

1. 先检查目标游戏目录
2. 再检查运行时目录
3. 如果运行时不存在，就先导入运行时
4. 配置本地翻译参数
5. 先做一次干运行安装
6. 确认无误后再正式安装
7. 启动本地翻译服务
8. 最后再启动游戏

这样出问题时更容易定位是哪一步出了偏差。

## 6. 命令行使用说明

## 6.1 检查游戏目录

用来判断目标目录是否像一个受支持的 `Unity Mono` 游戏：

```powershell
python main.py inspect-game "D:\Path\To\YourGame"
```

如果识别成功，输出里通常会包含这些信息：

- `game_dir`
- `exe_path`
- `data_dir`
- `managed_dir`
- `engine`
- `architecture`
- `is_unity_mono`

如果失败，常见原因是：

- 目录里没有 `.exe`
- 缺少 `*_Data`
- 缺少 `*_Data/Managed`
- 实际上不是 `Unity Mono`

## 6.2 检查运行时目录

用来判断当前 `runtime_root` 是否具备安装所需的基本结构：

```powershell
python main.py inspect-runtime --runtime-root "D:\Path\To\runtime\unity_mono"
```

当前代码会重点检查：

- `AutoTranslator/`
- `ReiPatcher/`
- `Managed/`

只要这三块缺任何一块，检查就会失败。

## 6.3 导入运行时

如果你手头有一个已经准备好的样例游戏目录，可以把其中需要的运行时内容提取到你自己的 `runtime_root`：

```powershell
python main.py import-runtime `
  --source-game "D:\Path\To\SourceGame" `
  --runtime-root "D:\Path\To\runtime\unity_mono"
```

这一步会做的事情包括：

- 重置目标 `runtime_root`
- 复制 `AutoTranslator`
- 复制 `ReiPatcher`
- 复制必要的 `Managed` 文件
- 复制 `Translators`
- 复制字体文件

注意：

- 这不是增量合并，而是会重置目标运行时目录
- 所以不要把别的手工文件混放到同一个 `runtime_root` 里

## 6.4 干运行安装

正式安装前，建议先执行一次干运行：

```powershell
python main.py install `
  --game "D:\Path\To\YourGame" `
  --runtime-root "D:\Path\To\runtime\unity_mono" `
  --target-language zh `
  --dry-run
```

干运行不会真的写入文件，但会返回计划执行的操作，例如：

- 复制 `AutoTranslator`
- 复制 `ReiPatcher`
- 复制 `Managed`
- 生成 `AutoTranslator\Config.ini`
- 生成 `ReiPatcher/*.ini`

这一步非常适合先确认路径有没有选错。

## 6.5 正式安装

确认没问题后，再去掉 `--dry-run`：

```powershell
python main.py install `
  --game "D:\Path\To\YourGame" `
  --runtime-root "D:\Path\To\runtime\unity_mono" `
  --target-language zh
```

正式安装时，工具会在目标游戏目录中写入或复制相关文件。

同时，它还会在目标游戏目录下写入安装清单：

```text
.private_translator/install_manifest.json
```

这个清单用于后续卸载和回滚。

## 6.6 卸载和回滚

如果你要撤销安装结果，可以执行：

```powershell
python main.py uninstall --game "D:\Path\To\YourGame"
```

当前实现会根据：

```text
.private_translator/install_manifest.json
```

来删除当前工具写入的文件，并尽量恢复备份内容。

如果这个清单丢了，卸载能力就会受影响。

## 7. 图形界面使用说明

如果你更习惯 GUI，可以启动桌面版：

```powershell
python main_gui.py
```

或者：

```powershell
game-translator-gui
```

图形界面更适合下面这些场景：

- 不想手写命令
- 想保存本地翻译配置
- 想通过按钮逐步操作
- 想直接观察日志输出

### 7.1 GUI 常用流程

推荐按这个顺序操作：

1. 选择目标游戏目录
2. 选择运行时目录
3. 如果需要，从样例目录导入运行时
4. 选择目标语言
5. 配置翻译模式
6. 保存翻译配置
7. 启动本地翻译服务
8. 先干运行安装
9. 再正式安装
10. 最后启动游戏

### 7.2 翻译模式

图形界面支持两种模式：

- `llm`
- `machine`

`llm` 模式通常需要你填写：

- API Base
- API Key
- Model
- System Prompt

`machine` 模式当前主要围绕：

- `deepl`
- `libretranslate`

### 7.3 保存配置后会发生什么

当你点击“保存翻译配置”后，程序会在本地创建：

```text
.private_translator/settings.json
```

在后续这些操作中，也可能触发静默保存：

- 启动翻译服务
- 安装
- 测试翻译

所以这个文件属于正常运行产物。

## 8. 本地翻译桥接服务

这个项目内置了一个本地 HTTP 翻译桥接服务。

默认配置下，它会使用类似下面的地址：

```text
http://127.0.0.1:14366/translate
```

这个服务的作用是：

- 接收游戏运行时发来的文本
- 根据当前配置决定走大模型还是普通机翻
- 返回翻译结果
- 把结果写入本地缓存

## 9. 本地缓存说明

翻译结果会优先使用本地缓存，减少重复请求。

缓存文件通常位于：

```text
.private_translator/translation_cache.json
```

如果你更换了：

- API Base
- 模型
- 机翻提供方

那历史缓存和当前结果可能不完全一致，这是正常现象。

## 10. 常见问题

### 10.1 为什么 `inspect-game` 失败

优先检查：

- 是否选到了真正的游戏根目录
- 目录下是否有主游戏 `exe`
- 是否存在 `*_Data/Managed`

这个项目不接受只选到 `Game_Data` 子目录的用法，通常要选游戏根目录。

### 10.2 为什么 `inspect-runtime` 失败

优先检查：

- `runtime_root/AutoTranslator` 是否存在
- `runtime_root/ReiPatcher` 是否存在
- `runtime_root/Managed` 是否存在

### 10.3 为什么安装前建议先 `--dry-run`

因为最常见的问题不是逻辑错误，而是路径选错：

- 把 `dist/PrivateGameTranslator` 误当成游戏目录
- 把错误目录当成运行时目录
- 目标目录不是 `Unity Mono`

先 `--dry-run` 可以先看到计划写入什么位置。

### 10.4 为什么卸载失败

常见原因有：

- `.private_translator/install_manifest.json` 不存在
- 安装后文件被手工改动过
- 游戏目录被移动过

### 10.5 为什么仓库里不建议提交本地配置和缓存

因为这些文件通常包含：

- API 配置
- 本地运行状态
- 翻译缓存结果

它们不属于源码本身，而且可能包含敏感信息或临时数据。

## 11. 仓库发布建议

如果你准备把这个项目作为公开 GitHub 仓库维护，建议至少做到：

1. 只公开源码、文档和必要脚本
2. 不公开用户本地配置和缓存
3. 不默认公开打包产物
4. 对第三方运行时的再分发权限单独确认
5. 在仓库首页明确“目前只支持 Unity Mono”
